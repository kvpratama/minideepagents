# Stage 04 — runner-managed log and index

## Stage summary

* **Current limitation:** the agent owns *every* writable file under
  `/wiki/`, including ones whose value depends on being deterministic
  and machine-maintained.
* **Naive fix that fails:** "tell the agent the schema of `log.md`" —
  models reformat, summarize, dedupe, and silently change conventions
  across runs.
* **Right fix:** runner owns `/log.md` and `/wiki/index.md`. Agent is
  *denied* write access to both, and the runner generates them after
  each phase.
* **New abstraction:** the **runner / agent contract** — a small set
  of file paths managed by deterministic Python instead of the model.

## What changed vs. stage 03

* Permissions gain two `deny` rules: `/log.md` and `/AGENTS.md`.
* Two new runner helpers, `append_log_entry()` and `refresh_index()`,
  do their work *outside* the agent invocation.
* The system prompt explicitly disclaims responsibility for those
  files — belt-and-suspenders with the deny rule.

## Failure demonstration

Stage 03's pressure was destructive evidence writes. Here the failure
is more insidious: drift. Imagine three ingests in a row where the
agent maintains `index.md` itself. After run one it categorizes by
"Topic". After run two it switches to "Theme". After run three it
forgets the older sections entirely and rewrites from scratch. The
catalog *looks* fine after each run; longitudinally it's noise.

The same applies to `log.md`. If the model writes "added some notes
on Ada" once and "✨ Updated the wiki ✨" another time, you lose the
ability to grep your own audit trail.

Running this stage shows the deterministic alternative: one entry
shape, one categorical index, both regenerated by ~30 lines of Python.

## Why simpler fixes fail later

* **Asking the agent to use a JSON schema for log entries.** Models
  comply ~95% of the time, and the 5% destroys downstream parsing.
* **Letting the agent edit `index.md` if it follows a template.**
  Templates accumulate cruft (deleted sections leave stubs, renamed
  categories duplicate). The whole point of an index is that it
  reflects the page set *right now*, which is exactly what a directory
  scan computes for free.
* **Storing logs in a database.** Reasonable in principle, but it
  splits the wiki's source of truth: the corpus is in git/Hub, the
  audit trail is elsewhere. Later stages push the whole workspace to
  Context Hub as one atomic unit, which is much easier if the log is
  a file in that workspace.

## Tradeoffs introduced

* The runner is no longer a thin entrypoint. It now owns derived
  state — index generation, log formatting. That code lives in
  `index.py` and `log.py` in the original repo, isolated for testing.
* The agent can't react to its own log. If it wants recency context,
  the prompt has to tell it to read recent `log.md` entries — and the
  runner has to keep those entries dense and parseable. This is
  exactly the constraint enforced by the `## [date] phase | key=value`
  header format in the real `log.py`.

## LangChain + LangGraph mapping

* Nothing graph-shaped yet. The runner code is plain Python that runs
  *around* the agent invocation.
* This is the first hint that "the agent" and "the system" aren't the
  same thing. Real production agents are usually a small orchestration
  shell with the model on the inside, not the other way around.

## Mentor mode

* **Aha:** the agent should never be in charge of artifacts whose
  value is "this is what really happened" or "this is what is
  currently here". Those are runner concerns.
* **Common mistake:** modelling `log.md` as a tool the agent calls
  ("`log_event(...)`"). Looks elegant; in practice the model forgets
  to call it, or calls it inconsistently across modes, and the audit
  trail still drifts.
* **Tempting alternative:** generating `index.md` *as a tool* and
  exposing it to the agent. Same failure: the agent decides when to
  refresh, instead of "after every applying phase, always".
